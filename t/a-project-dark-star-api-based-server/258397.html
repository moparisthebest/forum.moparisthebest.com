Project file and source is attached.

Project Dark Star is essentially aimed at people looking to create highly scalable and concurrent server software using the Java programming language. It is highly scalable, ensures serial access to any object that will require an update, and quite easy to use after reading the documentation. I believe a well-written server that depends on Dark Star would be superior to the RSCD framework.
The only downside is its means of persistence - all ManagedObject instances are persisted using synchronization, which allows objects to be passed from one server to another (thus scalability is possible).

It‚Äôs not an optimization. It‚Äôs simply a different design approach.

This is based on an unmodified RSCD v25 server.
Let‚Äôs say Player A is in an empty area. The method getViewedArea(x1, y1, x2, y2) is called several times (for items, game objects, players and npcs.
When getViewedArea is called by getNpcsInView() and getPlayersInView() the getViewedArea method performs 240 iterations in almost all cases. For each tile in the 2d array returned by the method, the ‚ÄúgetEntity()‚Äù method is invoked, and the list returned is passed as an arg to tempList.addAll(List).
For items and objects, 441 iterations occur, followed by the same adding of items and objects to a temp list.
So what happens when a player is in an absolutely empty area? All of that.
These methods are also heavily used; game updates depend on them, and the game updates occur each cycle. That‚Äôs a lot of work that can be completely avoided.

Let‚Äôs take a look at the alternative now.

The world is broken up into sections/areas. We choose area sizes based on expected activity. High-traffic map locations such as varrock are fragmented into small areas, whereas rarely used sections will have bigger areas (100100, 200200 etc).
Each area has a backing Channel object, which contains a collection of all ClientSession objects currently in the same area.
When a player gets close enough to a section, we join them to the area and its backing channel that they‚Äôre approaching.
They could be in as many as 4 Area objects at any given time, but more often than not they‚Äôd be in 2 Area objects.
When player updates occur we have them iterate over the collection of ClientSession objects - they grab the associated Player object and see if it‚Äôs within range. If it is then it updates the player‚Äôs client as required.
So say there are 150 players in an area in Varrrock. That‚Äôs a maximum of 150 iterations per player, and a maximum of 150 invocations of a withinDistance(Object) method.
As for items and game objects, I believe we could still use the ActiveTile object. The ActiveTile objects could be stored in a List, and iterated over. Even the most populated areas wouldn‚Äôt have too many tiles to iterate over. The tiles would be removed when the items on them are removed.

Performance wise I believe the solution I‚Äôm using is far more satisfying than RSCD‚Äôs approach. I think it will scale far better too - little, if any, processing power is used on players unless they are in the most populated areas - afterall, a player on their own in an area who could view another area with maybe one player would iterate over 2 Collections - the players from the area it‚Äôs in (ie, a single iteration), and the players from the area it can see a portion of (ie, another iteration).
