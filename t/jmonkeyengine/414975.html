Graham August 1, 2016, 3:29pm #7
IlikeArrays:

Who said impossible?

It appears that the OP was edited after Frell said impossible.

Also, imho JME isn‚Äôt amazing. The API isn‚Äôt very stable, as Taharok hinted at, and I personally don‚Äôt like some of the design. You don‚Äôt have a huge amount of choice if you‚Äôre going to use Java. There are some fairly nice C++ engines though - I‚Äôm doing a project in Ogre3D at the moment - so you might want to consider checking some of those out.

As for making a simple MMORPG, there are obviously two main components: the client and the server. Both can be kept fairly simple at first: just make the server keep track of all the players and broadcast updates to everyone (e.g. a player logged in, a player moved, etc). Eventually you‚Äôll want to extend it so the server only broadcasts updates about players that are near to the player receiving the updates, this involves keeping a local list of players within view of each client. The simple way to do this is a loop within a loop, check the updating sequence in some winterlove-based server for an example.

The client needs to also keep track of players it knows about, and since you are using JME actually getting the models displayed will be quite easy as you just need to add a node to the scene graph. You‚Äôll also eventually need to interpolate movement, since you can‚Äôt send movement updates in real time due to bandwidth constraints, but the movement still needs to appear smooth on the client. You can do that with some simple maths.

Obviously you‚Äôd need to also devise a protocol for communication, but that is also quite easy. Since you seem to be somewhat familiar with private servers, you could take an approach similar to Jagex or maybe you want to be more exotic. You just need to define a format in general for the packets (e.g. having a header consisting of an opcode byte and length word, followed by the packet data) as well as the formats in individual packets (e.g. adding a new player might have fields for the player id, name, x/y/z coords and rotation).

I wouldn‚Äôt worry so much about loading/saving players at first, that‚Äôs a somewhat boring detail when you are trying to get the more visual things working and if you‚Äôre capable of the above, a bit of file I/O or database stuff won‚Äôt be too taxing :p.
