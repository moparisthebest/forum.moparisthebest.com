And the problem that I can see, is that all of those names in that block, can only be processed if the name also contains ‚Äúaril‚Äù (which none of them probably do), which is probably what‚Äôs setting it to the incorrect value.

I would HIGHLY suggest improving how to get the correct values, rather than using strings. The reason strings were used, was because people thought that, since all daggers have the same speed, all item names ending in ‚Äúdagger‚Äù would be daggers, and the code could be simplified. Which is true for the most part, but then you get into situations like this where the blocks of code get condensed, and you end up repeating the conditions, returning different values each time.

The simplest, but most rigid way of changing this, would be to revert to the previous method of handling delays, by means of a switch statement, with the item ids being the cases.

switch (itemId) {
    case 1234:// You can use comments to show the name
        return 1;

    case 5678:// Some item
        return 2;

// etc...
}

But this requires a lot of hardcoding, and not really suggested. Another way of doing this is to load them from a specific file, specifying the delay when each item is parsed (items.cfg) which would be my suggestion.
